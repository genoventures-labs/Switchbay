# Switchbay 1.5 Roadmap

## Direction

Switchbay 1.5 moves Switchbay from a terminal coding shell toward a terminal-native operator layer for the developer's day.

The target is not a sci-fi assistant, a corporate chatbot, or a fake butler voice. Switchbay should feel like a grounded CLI homebase: aware of the current repo, recent work, local reminders, model lane, workspace state, and the next practical move.

Switchbay remains the workbench. Switchbay becomes the callsign for the operator experience inside it.

## Product Thesis

Developers do not only need code generation. They need a local cockpit that keeps context warm:

- What am I working on?
- What did we leave unfinished?
- What is on deck today?
- Which repo/session/lane am I in?
- What should I test before I ship?
- What did Switchbay learn about this workspace?
- What can be handled locally before spending cloud tokens?

Switchbay 1.5 should answer those questions with local state first, model calls only when they add value, and clear terminal UX.

## Accepted For 1.5

### Daily Board

Status: shipped.

Switchbay gets a lightweight daily board for short-lived tasks and reminders.

Storage:

- User config scope: `~/.switchbay/daily.json`
- JSON, not a database
- Cap at 5 active daily items
- Resets or archives by day
- No recurring/task-management bloat in the first pass

TUI behavior:

- On launch, Switchbay greets with a compact daily overview.
- Right rail can show today's board when space allows.
- Main feed greeting stays brief so it does not swallow the workspace.

CLI behavior:

```bash
switchbay agenda
switchbay task add "test Homebrew install"
switchbay task done 1
switchbay task clear
```

Slash behavior:

```text
/agenda
/task add test Homebrew install
/task done 1
/task clear
```

Conversational behavior:

```text
Switchbay, what's on my agenda?
Switchbay, remind me to test brew after lunch
Switchbay, mark the brew test done
Switchbay, clear today's board
```

Rules:

- Obvious agenda/task requests should route to local commands before model calls.
- If the board already has 5 active items, Switchbay should ask the user to complete or clear one.
- The daily board is operational memory, not long-term memory.

### Startup Overview

Status: shipped.

When the TUI opens, Switchbay shows a short overview assembled programmatically:

- Current workspace
- Current branch when available
- Runtime lane/provider/model
- Active daily board count
- Last session title or summary if available
- Stale or unfinished items when clearly known

Example:

```text
Switchbay
Good morning. It's Thursday, July 9.

Today: 2 open
Workspace: Switchbay on main
Lane: cloud/auto
Last: cloud provider JSON wiring
Next: run smoke checks or start 1.5 operator work
```

### Recall Commands

Status: partially shipped.

Switchbay should support practical recall without pretending to be a full personal knowledge system:

```text
Switchbay, what did we leave unfinished?
Switchbay, what did we change last time?
Switchbay, remind me what this repo does
Switchbay, what lane am I using?
```

First pass sources:

- Session persistence
- Workspace memory
- Workspace Knowledge index
- Git status
- Daily board
- Selected lane/model/provider config

### Local-First Intent Routing

Status: shipped for the current local state surfaces; keep expanding only when there is a concrete command/state source.

Before a model turn, Switchbay detects requests that can be answered from local state and handles them directly.

Local-first requests:

- agenda and daily board
- lane/provider/model status
- session list/resume/last-session summary
- workspace list/hop/current workspace
- memory list/search
- knowledge status/search
- engine/skill/plugin inventory
- MCP status/catalog
- git branch/status/diff summary
- config paths and active JSON settings

Conversational examples:

```text
Switchbay, what's my lane?
Switchbay, what models are available locally?
Switchbay, what workspace am I in?
Switchbay, show my sessions
Switchbay, what engines do I have?
Switchbay, is MCP on?
Switchbay, what's changed in git?
```

Rules:

- Route obvious local-state requests before calling a model.
- Show when Switchbay handled something locally.
- Fall back to the selected model lane only when the request needs reasoning, synthesis, or ambiguous interpretation.
- Keep local-first answers compact and command-like.
- Never invent state that is not present in local config, git, memory, sessions, or caches.

### Command Coach

Status: shipped for the core Switchbay workflows.

When the user asks how to do something Switchbay already supports, Switchbay answers with the exact slash or CLI command instead of sending the question to the model.

Examples:

```text
Switchbay, how do I switch to Ollama?
Switchbay, how do I sync engines?
Switchbay, how do I add a plugin?
Switchbay, how do I check cloud provider config?
```

Behavior:

- Answers from a curated command map, not model improvisation.
- Includes the exact slash command or CLI command.
- Can include one sentence of context when useful.
- Falls back to normal model routing only when no known command matches.

Goal: make Switchbay feel self-explaining without bloating the UI.

### Friction Radar

Status: shipped.

Switchbay can detect obvious blockers from local state and call them out quietly.

Signals:

- dirty git tree before release command
- tests failed in latest trace
- selected local provider unreachable
- missing cloud API key for selected provider
- MCP enabled but no config exists
- daily board full

Possible command:

```text
/radar
Switchbay, run radar
```

Behavior:

- Runs local checks only.
- Reports blockers, warnings, and clean signals separately.
- Does not mutate state.
- Shows the next suggested command when obvious.

### Quick Handoff Notes

Status: shipped.

Switchbay can create a concise handoff note for future Switchbay/Codex sessions.

Possible command:

```text
/handoff
Switchbay, write a handoff for next time
```

Output:

- current goal
- changed files
- tests run
- next best step
- known caveats

Behavior:

- Uses local state first: session, git, trace, memory, and daily board.
- Can optionally ask the active model to polish the summary when the user requests it.
- Saves to the session transcript and can be copied into memory if requested.

### Operator Mode Toggle

Status: shipped.

Keep the operator layer configurable so Switchbay can still launch as a quiet coding shell.

Possible config:

```json
{
  "operator": {
    "enabled": true,
    "startupOverview": true,
    "dailyBoard": true
  }
}
```

Rules:

- Operator mode defaults on for the 1.5 experience.
- Users can quiet startup/operator surfaces without disabling coding features.
- Local-first commands still work even when startup overview is disabled.

## Candidate Ideas

These are promising, but not yet accepted until we shape them.

### Continue-Where-We-Left-Off

Switchbay offers a compact next-step prompt based on the latest session, git diff, failed tests, or active daily board.

Possible commands:

```text
/continue
Switchbay, pick up where we left off
```

### Morning Loadout

Switchbay can offer a startup loadout that is more action-oriented than the normal overview.

Possible contents:

- current repo and branch
- dirty files count
- open daily tasks
- latest unfinished session
- suggested first command
- likely best lane for the next task

Possible commands:

```text
/loadout
Switchbay, give me the loadout
```

### Project Brief

Switchbay can generate or refresh a compact repo brief from README, `SWITCHBAY.md`, Workspace Knowledge, and recent sessions.

Possible commands:

```text
/brief
Switchbay, brief me on this repo
```

### End-Of-Session Wrap

Switchbay can summarize the current session into:

- Done
- Left open
- Suggested next
- Test state
- Daily board suggestions

Possible commands:

```text
/wrap
Switchbay, wrap this session
```

### Focus Lock

Switchbay can pin one lightweight focus for the current session so it can keep nudging the work back to the user's stated target.

Possible commands:

```text
/focus ship Homebrew 1.5 smoke
/focus clear
Switchbay, keep me focused on the release checklist
```

Behavior:

- Shows in the right rail when active.
- Included in startup overview while the session remains active.
- Local only; not a replacement for tasks.

## Later / 2.0 Candidates

- Recurring reminders
- Calendar or email connectors
- Richer personal preference memory
- Workspace-specific daily boards
- Cross-workspace agenda
- Release readiness assistant
- Plugin-provided operator panels
- Personal command macros
- Lightweight notification hooks
- Operator profiles by workspace
- Voice or notification surfaces

These should stay out of 1.5 unless they become obviously small.

## Non-Goals

- No heavy project management clone
- No fake personality layer that gets in the way
- No mandatory cloud model call on startup
- No hidden telemetry requirement
- No complex recurrence engine in 1.5
- No desktop app requirement

## 1.5 Smoke Checks

- `switchbay agenda` shows an empty daily board on a fresh config.
- `switchbay task add "test brew"` adds one item.
- `switchbay task done 1` marks it complete.
- TUI launch shows a compact overview without breaking layout.
- `/agenda` shows the same board as CLI.
- Conversational "Switchbay, what's on my agenda?" does not call the model.
- Conversational "Switchbay, remind me to test brew" adds a task.
- Conversational "Switchbay, what's my lane?" answers from local config without a model call.
- Conversational "Switchbay, what's changed in git?" answers from local git state without a model call.
- Conversational "Switchbay, how do I switch to Ollama?" answers with `/lane ollama` and the CLI equivalent.
- `/radar` reports missing keys, provider reachability, dirty git state, latest trace status, and daily board fullness.
- `/handoff` returns a compact handoff with goal, changed files, tests, next step, and caveats.
- Daily board refuses a sixth active item with a useful message.
- `bun test`, `bunx tsc --noEmit`, and `bun run build` pass.
