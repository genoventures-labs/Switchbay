# Quick Starts and Rules

Quick Starts and Rules are short instruction packets injected directly into Switchbay's system context at the start of every turn. They act like "read this first" cards — Switchbay sees them before doing any tool work, editing code, or selecting a lane.

They are not retrieved on demand. They are always present (up to 8 guides per session) so Switchbay can apply the right behavior automatically without you having to re-explain it.

---

## The Two Kinds

| Kind | Purpose |
|---|---|
| `quickstart` | How Switchbay should behave when using a specific tool, lane, or workflow |
| `rule` | Standing constraints Switchbay should always respect in this context |

Both kinds live in the same system, use the same file format, and are loaded from the same places. The only difference is how they're labeled and sorted in the prompt block.

---

## Built-In Guides

These ship with Switchbay and are always available. They are the baseline Switchbay reads on every session.

### `tool-use-quick-start` — Tool Use Quick Start
**Kind:** quickstart · **Triggers:** tools, files, edit, command, shell, tests, build

> The default method for local coding work.

- Read files and list directories before changing code.
- Prefer targeted `apply_patch` edits over full rewrites.
- Run focused verification after code changes: typecheck, tests, or build.
- Use approvals only for destructive, privileged, publishing, or broad external-impact actions.
- Keep tool summaries concise; put detailed traces in the tool panel, not the main answer.

---

### `web-research-quick-start` — Web Research Quick Start
**Kind:** quickstart · **Triggers:** web, latest, current, url, docs, source, verify online

> How Switchbay should use the guarded Web Engine.

- Use `web_fetch`, `web_headers`, or `web_links` only for explicit public URLs.
- Cite the URL when web-fetched facts affect the answer.
- Say when a page could not be fetched, was truncated, or lacked evidence.
- Do not imply live verification unless a web tool actually succeeded.

---

### `engine-bay-quick-start` — Engine Bay Quick Start
**Kind:** quickstart · **Triggers:** engine, engine bay, toolbox, tool call, run engine

> How Switchbay should use swappable engines.

- Use `list_engines` before assuming an engine exists.
- Use `list_engine_tools` before running an engine tool.
- Respect engine-level approval settings.
- If an engine is absent, explain how to sync or configure it instead of inventing tools.

---

### `mcp-lane-quick-start` — Switchbay MCP Quick Start
**Kind:** quickstart · **Triggers:** mcp, local tools, browser tools, plugin

> How Switchbay should handle Switchbay MCP bridge setup and failures.

- The MCP config defaults to `~/.switchbay/mcp.json`.
- Only trusted catalog integrations should be generated by Switchbay.
- Switchbay MCP bridge is enabled with `/mcp on` or `SWITCHBAY_TOOL_MODE=switchbay-mcp`.
- Cloud and local model lanes can both use Switchbay's normal tool bridge.
- Missing plugin handle errors usually come from a native MCP lane, not Switchbay's bridge.

---

### `local-first-rule` — Local First Rule
**Kind:** rule · **Triggers:** default, workspace, repo, local

> The standing operating constraint for all Switchbay sessions.

- Treat the current workspace as the active world.
- Do not cross into sibling repos, host services, or private infrastructure unless the user asks.
- Prefer repo patterns and existing helpers over new abstractions.

---

## How They Get Into the Prompt

Every session, `buildGuidesPromptBlock()` loads all guides, sorts them by priority, takes up to **8**, and injects them into the system prompt under:

```text
QUICK STARTS AND RULES (read before matching tool/lane work):
### Tool Use Quick Start (tool-use-quick-start; builtin/quickstart)
...
### Local First Rule (local-first-rule; builtin/rule)
...
```

Switchbay sees this block on every turn — it is not retrieved, not summarized, not dependent on the conversation history.

### Priority Order (highest first)

1. **Workspace** guides (`.switchbay/quickstarts/`, `.switchbay/rules/`)
2. **Plugin** guides (`.switchbay/plugins/<id>/guides/`)
3. **User** guides (`~/.switchbay/quickstarts/`, `~/.switchbay/rules/`)
4. **Built-in** guides (shipped with Switchbay)

If two guides share the same `id`, the higher-priority source wins. This means a workspace rule can override or replace a built-in rule entirely by using the same `id`.

## Adding Your Own

### User-Scoped (all repos)

```bash
~/.switchbay/quickstarts/my-method.md      # quick start
~/.switchbay/rules/my-rule.rule.md         # rule
```

### Workspace-Scoped (this repo only)

```bash
.switchbay/quickstarts/my-method.md
.switchbay/rules/my-rule.rule.md
```

### Via Plugin

```bash
.switchbay/plugins/<id>/guides/my-guide.md
```

### Via TUI

```text
/create-rule       Conversational rule builder (saves to user or workspace)
/quickstarts       List all active quick starts
/rules             List all active rules
```

The conversational builder in `/create-rule` asks what the rule is for, when it applies, and whether it should be user-global or workspace-local. It generates the frontmatter for you.

---

## File Format

```markdown
---
id: my-method
title: My Method
kind: quickstart
description: One-line description of what this guide covers.
triggers: [keyword, another keyword, scenario]
---

# My Method

## Applies When

- The user asks about X or Y.
- Switchbay is about to use tool Z.

## Method

- Step one.
- Step two.
- Step three.

## Guardrails

- Never do this.
- Always check that first.
```

### Frontmatter Fields

| Field | Required | Description |
|---|---|---|
| `id` | Yes | Unique slug. Used for deduplication across sources. |
| `title` | Yes | Human-readable name shown in `/quickstarts` and `/rules` |
| `kind` | Yes | `quickstart` or `rule` |
| `description` | Recommended | One-line summary shown in the guide list |
| `triggers` | Recommended | Keywords Switchbay matches to decide if this guide is relevant |

`triggers` are not currently used for retrieval filtering — all guides in the top 8 are injected. They exist to tell Switchbay and the user what situations the guide is intended for.

---

## Viewing Active Guides

```bash
switchbay --help   # shows guide summary in startup
```

TUI:

```text
/quickstarts       List all loaded quick starts with source and ID
/rules             List all loaded rules with source and ID
```

Both commands show where each guide came from (`builtin`, `user`, `workspace`, or `plugin`) so you can audit what Switchbay is reading.
