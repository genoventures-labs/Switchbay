# Switchbay 1.5 Roadmap

## Direction

Switchbay 1.5 moves Bay from a terminal coding shell toward a terminal-native operator layer for the developer's day.

The target is not a sci-fi assistant, a corporate chatbot, or a fake butler voice. Bay should feel like a grounded CLI homebase: aware of the current repo, recent work, local reminders, model lane, workspace state, and the next practical move.

Switchbay remains the workbench. Bay becomes the callsign for the operator experience inside it.

## Product Thesis

Developers do not only need code generation. They need a local cockpit that keeps context warm:

- What am I working on?
- What did we leave unfinished?
- What is on deck today?
- Which repo/session/lane am I in?
- What should I test before I ship?
- What did Bay learn about this workspace?
- What can be handled locally before spending cloud tokens?

Switchbay 1.5 should answer those questions with local state first, model calls only when they add value, and clear terminal UX.

## Accepted For 1.5

### Daily Board

Bay gets a lightweight daily board for short-lived tasks and reminders.

Storage:

- User config scope, likely `~/.switchbay/daily.json`
- JSON, not a database
- Cap at 5 active daily items
- Resets or archives by day
- No recurring/task-management bloat in the first pass

TUI behavior:

- On launch, Bay greets with a compact daily overview.
- Right rail can show today's board when space allows.
- Main feed greeting stays brief so it does not swallow the workspace.

CLI behavior:

```bash
switchbay agenda
switchbay task add "test Homebrew install"
switchbay task done 1
switchbay task clear
```

Slash behavior:

```text
/agenda
/task add test Homebrew install
/task done 1
/task clear
```

Conversational behavior:

```text
Bay, what's on my agenda?
Bay, remind me to test brew after lunch
Bay, mark the brew test done
Bay, clear today's board
```

Rules:

- Obvious agenda/task requests should route to local commands before model calls.
- If the board already has 5 active items, Bay should ask the user to complete or clear one.
- The daily board is operational memory, not long-term memory.

### Startup Overview

When the TUI opens, Bay shows a short overview assembled programmatically:

- Current workspace
- Current branch when available
- Runtime lane/provider/model
- Active daily board count
- Last session title or summary if available
- Stale or unfinished items when clearly known

Example:

```text
Bay
Morning. Here's the board.

Today: 2 open
Workspace: Switchbay on main
Lane: cloud/auto
Last: cloud provider JSON wiring
Next: run smoke checks or start 1.5 operator work
```

### Recall Commands

Bay should support practical recall without pretending to be a full personal knowledge system:

```text
Bay, what did we leave unfinished?
Bay, what did we change last time?
Bay, remind me what this repo does
Bay, what lane am I using?
```

First pass sources:

- Session persistence
- Workspace memory
- Workspace Knowledge index
- Git status
- Daily board
- Selected lane/model/provider config

## Candidate Ideas

These are promising, but not yet accepted until we shape them.

### Continue-Where-We-Left-Off

Bay offers a compact next-step prompt based on the latest session, git diff, failed tests, or active daily board.

Possible commands:

```text
/continue
Bay, pick up where we left off
```

### Operator Mode Toggle

Keep the operator layer configurable so Switchbay can still launch as a quiet coding shell.

Possible config:

```json
{
  "operator": {
    "enabled": true,
    "startupOverview": true,
    "dailyBoard": true
  }
}
```

### Project Brief

Bay can generate or refresh a compact repo brief from README, `SWITCHBAY.md`, Workspace Knowledge, and recent sessions.

Possible commands:

```text
/brief
Bay, brief me on this repo
```

### Local-First Intent Routing

Before a model turn, Bay detects requests that can be answered from local state:

- agenda
- lane/provider/model
- sessions
- workspace list/hop
- memory list/search
- knowledge search
- engines/skills/plugins
- git status

Goal: fewer unnecessary model calls, faster operator feel.

### End-Of-Session Wrap

Bay can summarize the current session into:

- Done
- Left open
- Suggested next
- Test state
- Daily board suggestions

Possible commands:

```text
/wrap
Bay, wrap this session
```

## Later / 2.0 Candidates

- Recurring reminders
- Calendar or email connectors
- Richer personal preference memory
- Workspace-specific daily boards
- Cross-workspace agenda
- Release readiness assistant
- Plugin-provided operator panels
- Voice or notification surfaces

These should stay out of 1.5 unless they become obviously small.

## Non-Goals

- No heavy project management clone
- No fake personality layer that gets in the way
- No mandatory cloud model call on startup
- No hidden telemetry requirement
- No complex recurrence engine in 1.5
- No desktop app requirement

## Implementation Slices

1. Add daily board store and CLI helpers.
2. Add slash/local command handling for agenda and tasks.
3. Add conversational intent parsing for obvious agenda/task/recall prompts.
4. Add TUI startup overview.
5. Add right-rail Today panel.
6. Add recall helpers for latest session, current lane, workspace, and git state.
7. Add docs and 1.5 smoke checklist.

## 1.5 Smoke Checks

- `switchbay agenda` shows an empty daily board on a fresh config.
- `switchbay task add "test brew"` adds one item.
- `switchbay task done 1` marks it complete.
- TUI launch shows a compact overview without breaking layout.
- `/agenda` shows the same board as CLI.
- Conversational "Bay, what's on my agenda?" does not call the model.
- Conversational "Bay, remind me to test brew" adds a task.
- Daily board refuses a sixth active item with a useful message.
- `bun test`, `bunx tsc --noEmit`, and `bun run build` pass.
